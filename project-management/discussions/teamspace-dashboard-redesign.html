Current UX Issues Identified

I've reviewed the current teamspace dashboard and here are my observations:

  1. Visual hierarchy problems:
    • The redundant logo creates confusion - users see branding twice which feels unprofessional
    • The welcome banner uses a red gradient that, while on-brand, doesn't provide clear visual hierarchy for a dashboard
    • The "Your Channels" section jumps straight into content without context
  2. Dashboard expectations:
    • Users landing here expect to see an overview of their entire operation
    • Currently it's just a channel picker, not a command center
    • No at-a-glance metrics means users must navigate into each channel to understand status

Proposed Layout Structure

+----------------------------------------------------------+ | WELCOME SECTION (Cleaner, no logo) | | "Welcome back, [Name]" + Quick stats summary | +----------------------------------------------------------+ | | | VITAL STATS SECTION | | +----------+ +----------+ +----------+ +----------+ | | | Total | | Total | | Videos | | Avg | | | | Views | | Subs | | This | | Watch | | | | -- | | -- | | Month | | Time | | | +----------+ +----------+ +----------+ +----------+ | | (Placeholders with "Coming Soon" / "Connect YouTube") | | | +----------------------------------------------------------+ | | | YOUR CHANNELS | | +----------------+ +----------------+ +-------------+ | | | Channel 1 | | Channel 2 | | + New | | | | [mini stats] | | [mini stats] | | Channel | | | +----------------+ +----------------+ +-------------+ | | | +----------------------------------------------------------+

Key UX Recommendations

  • Remove the logo - Sidebar handles branding; content area should focus on functionality
  • Personalised greeting - "Welcome back, [Name]" feels warmer than generic "Welcome to [Workspace]"

  • Card improvements - Channel cards could show mini-stats when available

Technical Implementation Analysis

TRON's layout proposal is solid. Let me break down the technical approach:

1. Remove Redundant Logo (Trivial)

// Simply remove these lines from teamspace-dashboard.tsx: {/* Branding header */} <div className={styles.branding}> <Image src="/streamline-studio-logo.png" ... /> </div>

And delete the corresponding .branding and .logo styles from the SCSS module.

2. Restructure Welcome Section

Currently the component receives teamspaceName but we could also pass the user's name. However, we already have access to the user via the trpc.user.me query in the app-shell. We could either:

  • Option A: Pass userName as a prop from the page server component
  • Option B: Make this a client component that fetches its own user data

I recommend Option A - keep the component flexible and data-fetching at the page level.

3. Stats Section Component Architecture

// New components needed: src/components/stats/ ├── stat-card.tsx // Reusable stat card component ├── stat-card.module.scss ├── stats-grid.tsx // Grid container for stat cards ├── stats-grid.module.scss └── index.ts // Usage in teamspace-dashboard.tsx: <StatsGrid> <StatCard label="Total Views" value={stats?.totalViews} placeholder="Connect YouTube" icon={<Eye />} /> <StatCard label="Subscribers" value={stats?.subscribers} placeholder="Coming Soon" icon={<Users />} /> ... </StatsGrid>

4. Data Model Considerations

For now, we're building placeholder UI. But I want to ensure we structure it properly for when YouTube API integration comes:

  • Stats will likely be aggregated across all channels in a teamspace
  • We'll need a youtubeIntegration table in the database
  • The StatCard component should handle: loading, error, placeholder, and data states

5. Styling Approach

I'll use the existing theme tokens. The current red gradient welcome banner could be refined to:

  • Lighter, more subtle gradient using --palette-grey-50 to --palette-grey-100
  • Or keep a subtle brand touch with --palette-red-50 as a border accent
  • Stats cards should use --color-surface-raised with --shadow-sm

Component Blueprint

Building on the Senior Dev's architecture, here's my proposed component structure:

StatCard Component Design

interface StatCardProps { /** Display label for the stat */ label: string; /** Current value (undefined = placeholder state) */ value?: number | string; /** Text shown when no value */ placeholder?: string; /** Icon component (lucide-react) */ icon?: ReactNode; /** Trend indicator: 'up' | 'down' | 'neutral' */ trend?: 'up' | 'down' | 'neutral'; /** Percentage change for trend */ trendValue?: number; /** Loading state */ isLoading?: boolean; /** Link to drill-down page */ href?: string; }

State Handling Strategy

  • Placeholder state: Shows "Coming Soon" or "Connect YouTube" with a subtle badge
  • Loading state: Skeleton animation using CSS (no external libraries)
  • Error state: "Unable to load" with retry option
  • Data state: Shows value with optional trend indicator

Refinement on Empty/Placeholder States

Feature Architect raises a good point about the "Connect YouTube" CTA. Let me elaborate on how to handle this well:

Placeholder UX Patterns

  • Don't just show dashes - That looks broken. Show a purposeful message.
  • Be aspirational - "See your channel analytics here" sets expectations positively
  • Provide a path forward - Even if the feature isn't ready, a "Coming Soon" badge builds anticipation

Recommended Empty State Content

// For the stats section when no YouTube integration exists: <div className={styles.statsPlaceholder}> <BarChart3 className={styles.placeholderIcon} /> <h3>Channel Analytics</h3> <p>Connect your YouTube channels to see aggregated stats across your entire operation.</p> <Badge variant="info">Coming Soon</Badge> </div>

Visual Treatment

  • Subtle dashed border around the stats area
  • 60% opacity on placeholder content
  • Single centred message rather than 4 empty cards
  • This feels intentional rather than incomplete

Accessibility Consideration

Ensure placeholder states have appropriate ARIA labels. Screen reader users should understand this is upcoming functionality, not a loading failure.
